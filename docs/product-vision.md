# Atlas Manager — Product Vision

## Document status

- Status: Draft
- Version: 0.1
- Author: Gustavo Pinto
- Last updated: 2026-07-18

## 1. Product summary

Atlas Manager is a self-hosted application for managing and automating a personal Linux homelab server.

It provides a central place to monitor the server, control applications and services, define availability schedules, manage containers and databases, execute backups, and coordinate power-management routines.

## 2. Problem

Managing a personal server usually requires several disconnected tools, manual terminal commands, configuration files, scheduled scripts, and knowledge about multiple infrastructure technologies.

Common operations such as checking service health, stopping an application, scheduling a container, creating a database backup, or configuring the next automatic power-on may require separate commands and manual verification.

This makes the server harder to operate, maintain, understand, and recover.

## 3. Target users

The initial target user is a developer or technology student who operates a personal Linux server or small homelab.

The user may host:

- personal APIs;
- portfolio applications;
- development databases;
- Docker containers;
- monitoring tools;
- experimental projects.

The first real user of Atlas Manager is its creator, Gustavo Pinto, who runs the Atlas server.

## 4. Value proposition

Atlas Manager centralizes common server-management and automation operations in one understandable application.

It helps users:

- understand the current state of their server;
- control individual applications without shutting down the whole machine;
- define when each service should be available;
- reduce unnecessary resource usage;
- automate backups and maintenance;
- operate Docker containers and databases through controlled workflows;
- manage server power schedules;
- keep a history of important operations and failures.

## 5. Product goals

Atlas Manager should:

1. Provide clear server and service health information.
2. Allow registered applications and services to be started and stopped.
3. Support independent availability schedules for each service.
4. Support PM2, Docker, systemd, and database integrations incrementally.
5. Automate safe shutdown and RTC wake-up routines.
6. Create and validate backups.
7. Record operational events and failures.
8. Expose administrative operations securely.
9. Provide both an API and, later, a web dashboard and CLI.
10. Remain understandable and useful as an educational open-source project.

## 6. Non-goals

Atlas Manager is not initially intended to:

- replace Portainer;
- replace Grafana or Prometheus;
- replace database clients such as DBeaver or MongoDB Compass;
- manage arbitrary shell commands;
- provide unrestricted access to Docker;
- manage large clusters or multiple data centers;
- replace enterprise infrastructure orchestration tools;
- act as a generic SQL editor;
- expose privileged operations without authentication and authorization;
- run the Express application as root.

## 7. Product principles

### Learning before automation

Important project logic should be understood and implemented by the project owner instead of being entirely generated by automated agents.

### Security by design

Privileged operations must use allowlists, minimal permissions, validation, auditing, and controlled adapters.

### Read before write

New integrations should begin with read-only monitoring before receiving permission to modify the server.

### Mock before production

Dangerous operations such as shutdown, container control, restoration, and service management should first have simulated implementations.

### Explicit over magical

The application should prefer visible and understandable behavior instead of hidden automation.

### Small and incremental releases

Each version should deliver a limited, testable, and documented capability.

## 8. Initial success criteria

The first stable version will be considered successful when it can:

- monitor the Atlas server and its main services;
- control registered PM2 applications;
- schedule individual application availability;
- monitor and control allowlisted Docker services;
- monitor supported databases and create logical backups;
- schedule safe shutdown and wake-up;
- maintain an operational event history;
- provide a protected administrative interface;
- recover from a documented deployment rollback;
- run reliably on the Atlas server.

## 9. Long-term vision

Atlas Manager may evolve into an open-source platform for developers who want a simple and educational way to manage a personal Linux server.

Its long-term direction may include:

- reusable service drivers;
- multiple supported database engines;
- notification integrations;
- richer observability;
- remote agents for additional servers;
- configuration import and export;
- backup destinations outside the local server;
- community-created adapters.

These possibilities do not belong to the initial release unless explicitly added to the roadmap.
